# OpenAlex 限流增强设计文档

## 1. 目标

本设计用于增强 OpenAlex 抓取器在共享版多方向池场景下的稳定性，降低因 `429 Too Many Requests` 导致整轮运行 `fetched=0` 的概率。

本轮目标仅包含：

- 将 OpenAlex 单次请求 `per_page` 从 `200` 调整为 `100`
- 对 `429` 响应增加指数退避重试
- 优先使用 `Retry-After` 响应头指导等待时长
- 在不同 `topic` 请求之间继续保留固定延迟
- 日志中补充限流相关响应头，提升可观测性

本轮不包含：

- 全局请求调度器
- 跨数据源统一限流器
- OpenAlex 付费套餐或 API key 接入
- 对 arXiv 限流逻辑的同步重构

## 2. 设计选择

本轮采用“最小增强型”方案，而不是一次性重做整个抓取调度层。

原因如下：

- 当前主要失败源已经明确是 OpenAlex 的 `429`
- OpenAlex 的问题集中在请求节奏与分页大小，不需要引入复杂架构才能先缓解
- 共享版引入多方向池后，请求次数增加，更需要低侵入但立刻见效的节流措施

因此本轮保持如下边界：

- 只修改 `fetchers/openalex.py`
- 只补 OpenAlex 相关测试
- 保持 `pipeline` 与主流程对失败的容错行为不变

## 3. 请求参数调整

本轮将 OpenAlex 的 `per_page` 上限从当前的 `200` 调整为 `100`。

原因：

- 与 OpenAlex 文档建议保持一致
- 降低单次请求压力，减少限流概率
- 在近 24 小时抓取场景下，`100` 对当前项目规模已经足够

实施后行为：

- 若外部配置传入更大值，也在内部截断为 `100`
- 仍通过 `per-page` 参数发起请求

## 4. 429 重试策略

当 OpenAlex 返回 `429 Too Many Requests` 时，抓取器不应立即失败，而是执行有限次重试。

推荐策略：

- 最大重试次数：`3`
- 优先读取响应头中的 `Retry-After`
- 若 `Retry-After` 缺失或非法，则使用指数退避：
  - 第 1 次重试：等待 `5s`
  - 第 2 次重试：等待 `10s`
  - 第 3 次重试：等待 `20s`

重试后若仍失败：

- 抛出异常给上层
- 由现有 `pipeline` 按当前逻辑记录 warning 并继续其他数据源或其他方向池

这样保持现有系统的“单源失败不拖垮全局”原则。

## 5. topic 间请求节奏

当前 OpenAlex 抓取器已经在多个 `filter` 间保留固定等待。

本轮保持该行为，并明确其角色：

- 它是“常规节流”
- 与“429 后的退避重试”互补

推荐继续保留：

- 每个 `filter` 正常请求之间等待 `2~3s`

这样可以避免共享版多方向池连续命中 OpenAlex 时，短时间内请求过密。

## 6. 日志与可观测性

本轮需增强 OpenAlex 抓取失败日志，至少记录：

- 当前 `filter`
- 当前重试次数
- 下次等待秒数
- `Retry-After`
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`

推荐日志形态类似：

```text
OpenAlex rate limited for filter=robotics, attempt=2/3, retry_after=10, remaining=0, waiting=10s
```

如果是非 `429` 错误：

- 保持当前异常语义
- 不伪装成限流问题

这样后续就能区分：

- 真的被限流
- SSL/网络波动
- 某个方向池配置过于激进

## 7. 错误处理边界

本轮只增强 `429` 处理。

具体规则：

- `429`：
  - 走限流重试路径
- 非 `429` 的 `requests.HTTPError`：
  - 维持当前行为，直接抛出
- `SSLError`、`ReadTimeout`、连接错误：
  - 维持当前行为，直接抛出

这样可以避免把所有错误都误归类成“限流”，导致问题排查困难。

## 8. 测试策略

重点补充 `tests/test_openalex_fetcher.py`。

至少覆盖以下场景：

### 8.1 正常请求参数

覆盖以下行为：

- `per_page` 被限制在 `100`
- 仍正确拼接 `filter` 和 `mailto`

### 8.2 `Retry-After` 驱动的重试

覆盖以下行为：

- 第一次请求返回 `429`
- 响应头中存在 `Retry-After`
- 抓取器按头部值等待并重试
- 第二次请求成功返回结果

### 8.3 无 `Retry-After` 时指数退避

覆盖以下行为：

- 连续 `429`
- 无 `Retry-After`
- 等待时间依次为 `5 -> 10 -> 20`

### 8.4 超过最大重试后失败

覆盖以下行为：

- 连续超过最大次数仍为 `429`
- 最终抛出异常

### 8.5 非 `429` 错误不走限流逻辑

覆盖以下行为：

- 如 `500` 或其他 `HTTPError`
- 不执行限流退避
- 直接抛出异常

## 9. 实施范围控制

本轮明确延后：

- 为 arXiv 增加统一调度器
- 为所有数据源统一输出 rate-limit 指标
- 将限流参数暴露到 YAML 配置
- 失败任务重试队列

本设计的交付标准是：OpenAlex 在遇到 `429 Too Many Requests` 时，能够以 `per_page=100`、带有限次重试和退避等待的方式继续尝试；若最终仍失败，系统保持当前容错语义，同时日志中可清晰看到限流上下文。
