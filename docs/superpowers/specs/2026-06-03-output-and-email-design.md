# 结果导出与邮件推送设计文档

## 1. 目标

本设计为当前文献抓取系统补上“结果导出 + 邮件推送”能力，使系统能够在抓取、匹配、开放获取增强和入库完成后，将本次需要推送的相关论文整理成可读摘要，并通过 SMTP 邮件发出。

本轮实现目标仅包含：

- 从主流程中拿到“本次命中的论文”
- 过滤掉已经推送过的论文
- 生成一份可读的 Markdown 文本摘要
- 通过 SMTP 发送邮件
- 发送成功后记录 `push_log`

本轮不包含：

- HTML 富文本邮件
- 多收件人策略和分组推送
- 附件导出（如 CSV、PDF）
- Web 查看页面
- 失败重试队列

## 2. 设计选择

采用“主流程产出结果 + 独立通知层处理导出和邮件”的方案，而不是把渲染和 SMTP 逻辑直接塞进 `pipeline`。

具体原因：

- `pipeline` 继续专注于抓取、筛选、增强、入库
- 导出格式和邮件发送属于输出层职责，单独测试更清晰
- 后续如果要增加本地文件导出、钉钉/企业微信/Slack 推送，可以直接复用同一份摘要内容

因此本轮保持如下边界：

- `pipeline` 返回结构化结果
- `email_renderer.py` 只负责把论文列表渲染为文本
- `smtp_sender.py` 只负责发送
- `repositories.py` 负责读取和写入 `push_log`
- `cli.py` / `main.py` 负责编排这些步骤

## 3. 主流程位置

本轮新增的数据流为：

1. `run_pipeline()` 完成抓取、匹配、OA 增强、入库
2. 主流程拿到本次命中的论文列表 `matched_records`
3. 使用 `PushLogRepository` 过滤掉已经推送过的论文
4. 若还有待推送论文：
   - 生成 Markdown 摘要
   - 调用 SMTP 发送邮件
   - 发送成功后写入 `push_log`
5. 若没有待推送论文：
   - 不发送空邮件
   - 返回“0 papers to push”之类结果

这样做的好处是：

- 推送去重发生在最终发送前，语义最清晰
- 数据库里仍保留全部抓取和匹配结果，不因为是否已推送而影响入库
- 邮件正文和本地导出可共用同一份渲染内容

## 4. 模块边界

### 4.1 `processing/pipeline.py`

负责：

- 执行抓取、关键词匹配、OA 增强
- 入库 `papers`
- 返回结构化 `PipelineResult`

本轮应扩展 `PipelineResult`，至少包含：

- `fetched_count`
- `matched_count`
- `matched_records`

其中：

- `matched_records` 表示当前运行中命中的论文列表
- 该字段只供后续导出和邮件推送使用

### 4.2 `notify/email_renderer.py`

负责把待推送论文列表渲染为 Markdown 文本。

输入：

- `records: list[PaperRecord]`

输出：

- 邮件正文字符串

渲染内容至少包含：

- 总数
- 每篇论文的标题
- 作者
- 来源
- 发布时间
- DOI
- 命中的关键词组
- 摘要节选
- 落地页链接
- PDF 链接或“需订阅”说明

MVP 阶段使用纯文本 / Markdown 即可，不做 HTML 模板。

### 4.3 `notify/smtp_sender.py`

负责真正发送邮件。

输入至少包含：

- SMTP 配置
- 收件人
- 邮件标题
- 邮件正文

发送器只关心：

- 建立 SMTP 连接
- 登录
- 发送

若发送失败：

- 抛出异常
- 由上层编排逻辑决定是否记录 `push_log`

### 4.4 `storage/repositories.py`

现有 `PushLogRepository` 需要扩展：

- 查询某篇论文是否已推送
- 批量记录本次成功推送的论文

MVP 阶段不要求复杂批量接口，只要主流程能够逐条判断和逐条写入即可。

### 4.5 `main.py` / `cli.py`

负责：

- 读取配置
- 调用 `run_pipeline()`
- 调用导出和发送逻辑
- 打印最终摘要信息

CLI 本轮仍保留单一命令：

```bash
python -m paper_crawler.cli run --config config
```

暂不新增 `notify` 子命令，避免超出 MVP 范围。

## 5. 推送去重规则

MVP 阶段采用最简单可靠的规则：

- 只要某篇论文的 `paper_id` 已存在于 `push_log`，就视为“已推送”
- 本次运行中已推送成功的论文，发送成功后写入 `push_log`
- 发送失败时，不写入 `push_log`

这意味着：

- 同一论文不会重复出现在后续邮件中
- 即便程序多次运行，也不会重复推送历史已发内容
- 若 SMTP 发送失败，下次运行仍有机会再次尝试发送这些论文

## 6. 邮件内容结构

MVP 阶段邮件正文采用 Markdown 风格文本，结构如下：

```text
# 今日光电子相关论文摘要

共 N 篇新论文。

## 1. 论文标题
- Authors: ...
- Source: ...
- Published At: ...
- DOI: ...
- Matched Keywords: ...
- Access: ...
- Landing URL: ...
- PDF URL: ... / 需订阅
- Abstract: ...
```

设计原则：

- 先保证可读性
- 保证正文在邮件客户端中即使不渲染 Markdown 也能读懂
- 后续若切换 HTML，可保留相同字段布局

## 7. 配置需求

当前 `config.yaml` 中已经有基础 SMTP 配置：

- `host`
- `port`
- `username`
- `from_address`

本轮还需要补充或明确：

- 收件人地址，例如 `to_address`
- 是否启用 TLS（建议默认启用）

SMTP 密码仍应通过环境变量注入，例如：

- `SMTP_PASSWORD`

不得硬编码进源码。

## 8. 返回值与 CLI 输出

主流程除了打印计数，还应输出更明确的结果摘要，例如：

- 抓取数量
- 命中数量
- 待推送数量
- 是否发送成功

例如：

```text
Pipeline finished: fetched=23, matched=7, to_push=5, email_sent=yes
```

若没有待推送论文：

```text
Pipeline finished: fetched=23, matched=7, to_push=0, email_sent=no
```

这样可以直接用于 cron 日志查看。

## 9. 失败与容错

本轮需遵守以下原则：

- 抓取失败不应中断整体流程，已在现有逻辑中实现
- 邮件发送失败不应导致 `papers` 入库回滚
- 邮件发送失败时不写 `push_log`
- 没有待推送论文时不发送空邮件

也就是说：

- 抓取与入库是主链路
- 邮件发送属于后处理
- 后处理失败应被清晰记录，但不破坏已完成的数据入库

## 10. 测试策略

测试分三层：

### 10.1 渲染器单元测试

覆盖以下行为：

- 正确渲染单篇论文
- 正确渲染多篇论文
- 命中关键词、PDF 链接和订阅标记正确显示

### 10.2 SMTP 发送器单元测试

覆盖以下行为：

- 正确构造邮件主题和正文
- 正确使用 SMTP 配置
- 发送接口被调用

MVP 阶段使用注入式假对象测试，不直接连真实 SMTP。

### 10.3 主流程集成测试

覆盖以下行为：

- `PipelineResult` 返回 `matched_records`
- 已推送论文不会再次发送
- 待推送论文发送成功后写入 `push_log`
- 无待推送论文时不发送邮件
- 发送失败时不写 `push_log`

## 11. 实施范围控制

本轮明确延后：

- HTML 邮件模板
- 富样式排版
- 附件导出
- 多通道推送
- 发送失败后的自动重试
- 批量合并历史未推送论文

本设计的交付标准是：系统完成一次抓取后，能够把本次新的相关论文整理成一份邮件正文，成功发送给目标收件人，并避免重复推送已经发过的论文。
