# OpenAlex 限流增强设计文档（v2）

## 1. 目标

本设计用于提升 OpenAlex 抓取器在免费配额场景下的稳定性，降低共享版多方向池运行时因 `429 Too Many Requests` 导致整轮结果为空的概率。

本轮目标仅包含：

- 将 OpenAlex 单次请求 `per_page` 从 `200` 调整为 `100`
- 遇到 `429` 时执行有限次重试
- 优先读取 `Retry-After` 响应头
- 在缺少 `Retry-After` 时使用指数退避
- 在日志中输出限流相关响应头，便于观察剩余配额

本轮不包含：

- 多账号/多身份轮换同一免费接口
- 规避同一服务的配额限制
- 统一重构所有抓取器的调度层
- 为 OpenAlex 接入付费套餐

## 2. 设计选择

本轮采用“最小增强型限流处理”方案，而不是重做全局抓取调度器。

原因如下：

- 当前日志已经明确显示 OpenAlex 主要失败模式是 `429`
- 现有 `OpenAlexFetcher` 结构简单，适合局部增强
- 多方向池共享版会增加请求次数，更需要低侵入节流策略

因此本轮保持如下边界：

- 只修改 `src/paper_crawler/fetchers/openalex.py`
- 只补 `tests/test_openalex_fetcher.py`
- 保持 `pipeline` 对单源失败的容错语义不变

## 3. 请求参数与重试策略

### 3.1 请求参数

OpenAlex 请求参数调整为：

- `per_page=100`

即使外部配置传入更大值，也在抓取器内部截断为 `100`。

### 3.2 重试策略

当 OpenAlex 返回 `429` 时：

- 最大重试次数：`3`
- 若响应头存在 `Retry-After` 且可解析：
  - 按该值等待
- 否则：
  - 第 1 次重试等待 `5s`
  - 第 2 次重试等待 `10s`
  - 第 3 次重试等待 `20s`

若超过最大次数仍为 `429`：

- 直接抛出异常
- 由现有 `pipeline` 记录 warning 并继续处理其他数据源或其他 `topic`

### 3.3 常规节流

在多个 `filter` / `topic` 间继续保留固定延迟：

- 推荐 `2~3s`

这部分作为常规节流，与 429 退避重试互补。

## 4. 日志与可观测性

当 OpenAlex 命中限流时，日志至少记录：

- 当前 `filter`
- 当前重试次数
- 等待秒数
- `Retry-After`
- `X-RateLimit-Limit`
- `X-RateLimit-Remaining`

推荐日志形式：

```text
OpenAlex rate limited for filter=robotics, attempt=2/3, retry_after=10, remaining=0, waiting=10s
```

如果是非 `429` 错误：

- 保持当前异常行为
- 不伪装成限流问题

这样后续可以快速区分：

- 配额耗尽
- SSL/网络波动
- 某个 topic 请求策略过于激进

## 5. 测试策略

重点补 `tests/test_openalex_fetcher.py`，至少覆盖：

### 5.1 正常请求参数

- `per_page` 被限制在 `100`
- `mailto` 和 `filter` 仍正确拼接

### 5.2 `Retry-After` 驱动重试

- 第一次请求返回 `429`
- 响应头中存在 `Retry-After`
- 抓取器按该值等待后重试
- 第二次请求成功

### 5.3 指数退避

- 连续 `429`
- 缺少 `Retry-After`
- 等待顺序为 `5 -> 10 -> 20`

### 5.4 超过最大重试后失败

- 连续多次 `429`
- 超过最大次数后抛出异常

### 5.5 非 `429` 错误保持原语义

- 如 `500`、SSL 错误、超时
- 不走限流重试路径

## 6. 实施范围控制

本轮明确延后：

- 多数据源兜底调度器
- arXiv 与 OpenAlex 的统一限流层
- 将退避参数暴露到 YAML
- 基于配额头的动态节流器

本设计的交付标准是：OpenAlex 在遇到 `429` 时，能够以 `per_page=100`、有限次重试和退避等待的方式继续尝试，并把限流上下文写入日志；若最终仍失败，系统保持当前单源失败容错语义。
